# 人工复核页历史相似案例真实数据替换规范

## 背景
- 人工复核页的“历史相似案例”区域当前写死了两条演示文案。
- 这些内容不来自真实数据，属于误导性展示。

## 目标
- 移除人工复核页中的静态假案例。
- 复用现有 `/api/inner-data?view=history-cases` 返回的真实 `review_cases` 聚合结果。
- 当用户暂无已完成复核记录时，显示明确空态。

## 约束
- 只处理 UI 模块中的人工复核页面。
- 不新增接口，不改接口字段，不跨模块重构。
- 保持现有页面样式结构，仅替换该区域的数据来源。

## 实施方案
1. 为“历史相似案例”列表增加独立容器 ID。
2. 删除写死的两条演示卡片，改为加载占位。
3. 页面加载后请求 `/api/inner-data?view=history-cases`。
4. 将最近的真实历史复核记录渲染到该区域。
5. 若无数据，显示“暂无历史复核案例”空态提示。

## 验收标准
- 页面中不再出现 “营销活动导致的连接耗尽”“凌晨报表任务占用过多连接”等假文案。
- 历史相似案例区域只显示真实历史复核数据或空态。
- 现有人工复核主流程和待复核队列不受影响。
