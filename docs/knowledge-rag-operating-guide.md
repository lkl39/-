# 知识库与 RAG 运行指南

## 目标

这套系统里，规则库、知识库、RAG、人工复核分别负责不同环节：

- 规则库：发现异常或可疑模式
- 知识库：解释异常、提供根因和解决方案
- RAG：从知识库中取最相关的少量上下文给模型
- 人工复核：纠偏、补知识、沉淀漏报案例

## 四类知识源怎么分工

### 1. 主知识库 `knowledge_base`

作用：

- 作为默认主召回源
- 存放高频、稳定、可复用的异常案例

适合放入：

- Nginx `502/504`
- 数据库连接数耗尽
- 慢查询
- Java OOM
- 磁盘满
- DNS 解析失败
- 端口拒绝
- 配置缺失

### 2. 历史漏报库 `historical_missed_cases`

作用：

- 保存人工复核确认过的漏报案例
- 在同类问题再次出现时优先召回

适合放入：

- 之前规则没命中，但人工确认是有效异常
- 模型曾误判，人工纠正后仍有复用价值

### 3. 运维经验库 `ops_experience_library`

作用：

- 提供排查路径和处置思路
- 在案例库不够时补上下文

适合放入：

- “504 超时排查顺序”
- “磁盘满止损步骤”
- “权限失败排查清单”

### 4. 安全候选池 `security_knowledge_candidates`

作用：

- 承接外部安全文档、规则库、案例库
- 先映射，再筛选，再决定是否晋升

不应直接作为主知识库全量来源。

## RAG 实际怎么工作

当前主链路已经固定为：

`元数据过滤 -> 向量粗召回 -> 关键词辅助过滤 -> 重排 -> 返回 top3~5`

### 1. 元数据过滤

先按以下条件缩小范围：

- `error_type`
- `source_type`
- `verified`
- `knowledge_layer`

### 2. 向量粗召回

从三层知识库中找语义最像的条目。

### 3. 关键词辅助过滤

再用标题、关键词、日志片段、根因、解决方案过滤噪声。

### 4. 重排

优先级如下：

1. 历史漏报库
2. 主知识库
3. 运维经验库

同时考虑：

- `verified`
- `error_type` 对齐
- `source_type` 对齐
- 向量相似度
- 关键词得分
- `priority`

### 5. 返回给模型

最终只返回 `top3~5` 条，避免给模型灌太多噪声。

## 日常维护流程

### 新异常进入系统后

1. 规则先做命中或分类
2. RAG 从三层知识库拿上下文
3. 模型输出根因和方案
4. 人工复核确认是否准确

### 如果输出不准

按情况补库：

- 高频通用案例：补到 `knowledge_base`
- 曾经漏掉的案例：补到 `historical_missed_cases`
- 偏排障经验：补到 `ops_experience_library`
- 外部安全资料：先补到 `security_knowledge_candidates`

## 晋升标准

候选知识要进入主知识库，至少满足：

1. 能解释日志
2. 有明确根因
3. 有可执行解决方案
4. 能稳定映射到当前 8 大异常类
5. 不与已有主知识高度重复

## 当前推荐策略

### 主知识库优先补哪些

优先顺序：

1. `database_error`
2. `timeout`
3. `service_error`
4. `network_error`
5. `configuration_error`
6. `resource_exhaustion`
7. `permission_error`

### 安全候选池怎么用

建议每次只晋升一小批：

- 先筛高置信度
- 再筛与异常诊断直接相关
- 再补 embedding
- 再观察 RAG 结果质量

不要整池一次性推入主知识库。

- 每周复盘一次漏报库
- 每批晋升后观察实际召回结果
- 低价值或误映射条目不要直接推进主库
