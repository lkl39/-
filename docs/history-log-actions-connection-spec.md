# 历史日志行内动作对接规范

## 背景
- `/dashboard/tasks` 已迁入真实 Next 页面，但用户反馈历史日志存档页的行内 `下载`、`删除` 按钮仍表现得像假入口。
- 当前后端已经存在可复用能力：
  - `POST /api/inner-data` 的 `history-download`
  - `POST /api/inner-data` 的 `history-delete`
  - `app/logs/actions.ts` 中已有日志删除相关复用逻辑

## 目标
1. 让历史日志表格中的 `下载`、`删除` 按钮具备真实可感知的动作反馈。
2. 保持现有字段契约不变，不改日志表结构与接口入参命名。
3. 只处理 `历史日志存档` 模块，不扩散到规则、RAG、模型层。

## 范围
- `components/dashboard/pages/tasks/tasks-page.tsx`
- 允许复用现有日志动作层，但不重写整套日志管理流程

## 实施要求
1. 下载动作继续复用 `history-download`，但前端不能再依赖容易被浏览器拦截的弹窗方式触发下载。
2. 删除动作成功后，除了移除当前列表项，还要刷新页面数据，避免右侧概览和分页状态残留旧值。
3. 对没有可下载文件的行，前端要体现出不可下载状态，避免继续误导点击。
4. 失败时给出明确提示，避免用户误判为按钮无效。

## 非目标
- 不修改 `logs`、`analysis_results`、`review_cases` 等数据字段。
- 不重做历史日志页整体布局。
- 不删除兼容层或静态目录。

## 验收标准
1. 点击 `下载` 后能实际触发文件下载，或明确提示失败原因。
2. 点击 `删除` 后对应行消失，并且页面刷新后概览数据与分页状态保持一致。
3. 没有 `storagePath` 的日志不会继续显示为可下载假按钮。
4. `eslint` 通过。
