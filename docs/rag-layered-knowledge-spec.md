# RAG 分层知识库规范

## 背景

当前系统的 RAG 仍以 `knowledge_base` 单库为主，`historical_missed_cases` 只是预留表，没有真正进入主检索链路。

这会导致几个问题：

- 知识条目字段不统一，难以做稳定的元数据过滤
- 历史漏报案例无法在检索阶段优先命中
- 知识量增大后，单库混合检索会逐步退化
- 运维经验类知识与异常案例类知识混在一起，不利于重排和去重

## 目标

1. 建立三层知识库
2. 统一知识条目模板
3. 把历史漏报库正式接入主 RAG 链路
4. 将检索流程固化为：
   `元数据过滤 -> 向量检索粗召回 -> 关键词辅助过滤 -> 重排 -> 返回 top3~5`
5. 为后续知识量增长预留分层、分库、缓存、聚类去重、低频归档策略

## 分层知识库

### 1. 异常案例库

职责：

- 保存已经沉淀的标准异常案例
- 覆盖高频、稳定、复用价值高的问题
- 作为默认主知识源

当前落地：

- 继续复用 `public.knowledge_base`
- 在现有字段基础上补齐统一模板字段

### 2. 历史漏报库

职责：

- 保存人工复核确认的漏报案例
- 在同类问题再次出现时优先召回
- 作为“防重复漏报”的高优先级知识源

当前落地：

- 继续复用 `public.historical_missed_cases`
- 扩展到统一模板字段
- 正式进入 RAG 主检索链路

### 3. 运维经验库

职责：

- 保存操作经验、排查经验、运行手册、处置经验
- 不一定是严格的异常案例，但对定位和修复有高价值
- 在案例库不足时补充上下文

当前落地：

- 新增 `public.ops_experience_library`

## 统一知识条目模板

三层知识库统一使用以下语义字段：

| 字段 | 说明 |
| --- | --- |
| `title` | 条目标题 |
| `error_type` | 主异常类型，统一为 8 大类编码 |
| `keywords` | 关键词或标签 |
| `log_excerpt` | 典型日志片段 |
| `root_cause` | 根因说明 |
| `solution` | 处理建议 |
| `source_type` | 来源域，例如 `nginx / postgres / application / system / custom` |
| `verified` | 是否人工确认 |
| `updated_at` | 最近更新时间 |

补充字段：

| 字段 | 用途 |
| --- | --- |
| `embedding` | 向量检索 |
| `archived_at` | 低频归档 |
| `cluster_key` | 聚类去重 |
| `source_label` | 知识来源说明 |
| `priority` | 重排优先级 |

## 外包裹式兼容策略

为了避免大面积重写现有知识库模块，本轮采用外包裹式重构：

1. 保留 `knowledge_base` 现有字段
2. 通过新增字段补齐统一模板
3. 保留 `historical_missed_cases` 现有能力，但升级为正式漏报库
4. 新增统一视图 `rag_knowledge_entries`
5. RAG 层优先改为读取统一视图，不强行改前端知识管理页

## 检索流程

### Step 1. 元数据过滤

先按以下条件收窄候选范围：

- `error_type`
- `source_type`
- `verified = true` 优先
- `archived_at is null`
- 分层来源：
  - `historical_missed`
  - `exception_case`
  - `ops_experience`

过滤原则：

- 历史漏报库优先保留同 `error_type`、同 `source_type` 条目
- 异常案例库作为主召回来源
- 运维经验库作为补充来源

### Step 2. 向量检索粗召回

对每一层做向量粗召回，再汇总：

- 历史漏报库：优先级最高，召回数量更小但权重更高
- 异常案例库：主召回池
- 运维经验库：补充召回池

### Step 3. 关键词辅助过滤

基于以下字段做 lexical 过滤：

- `title`
- `keywords`
- `log_excerpt`
- `root_cause`
- `solution`

目的：

- 过滤只在向量上接近但语义不稳定的噪声条目
- 对高频异常和严格分类做更强约束

### Step 4. 重排

重排信号按优先级组合：

1. 是否来自历史漏报库
2. 是否 `verified = true`
3. `error_type` 是否对齐
4. `source_type` 是否对齐
5. 向量相似度
6. 关键词匹配分
7. `priority`
8. 去重后摘要质量

### Step 5. 返回 top3~5

最终只返回 `top3~5` 给模型：

- 优先保证来源多样性
- 避免同一 cluster 的重复条目
- 优先返回人工确认、近期更新的高质量条目

## 历史漏报库建设原则

### 入库来源

当人工复核确认以下情况时，沉淀到历史漏报库：

- 规则未命中但最终确认是异常
- 当前知识库未能给出有效支持
- 模型结论明显偏离，人工修正后有复用价值

### 入库内容

至少写入：

- `title`
- `error_type`
- `keywords`
- `log_excerpt`
- `root_cause`
- `solution`
- `source_type`
- `verified = true`
- `updated_at`

### 检索优先级

历史漏报库在主链路中应高于普通案例库：

- 同类问题优先返回漏报案例
- 同一个漏报模式再次出现时，不允许继续被完全忽略

## 知识库内容增多后的优化方案

### 1. 分层

- 异常案例、漏报案例、运维经验分层
- 不同层采用不同权重和召回数量

### 2. 分库

- 高频异常案例与经验知识分开存储
- 历史漏报独立存储，避免被普通知识淹没

### 3. 缓存

- 以 `error_type + source_type + normalized query` 作为缓存键
- 缓存最近高频检索结果

### 4. 聚类去重

- 对标题、日志片段、根因做聚类
- 相似条目合并为同一 `cluster_key`
- 重排阶段只保留每个 cluster 的最佳条目

### 5. 低频归档

- 长期未命中、未验证、低质量条目进入归档
- 归档条目默认不参与主召回

### 6. 人工确认案例优先

- `verified = true` 的条目高于未确认条目
- 历史漏报中的人工确认条目高于普通经验条目

## 验收标准

1. 三层知识库结构齐备
2. 三层条目模板统一
3. RAG 主链路使用统一检索流程
4. 返回结果控制在 top3~5
5. 历史漏报案例可优先进入检索结果
6. 人工复核确认的漏报能自动沉淀到漏报库
